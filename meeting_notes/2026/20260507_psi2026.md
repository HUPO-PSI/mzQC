## mzQC manuscript

The first session focused on advancing the mzQC manuscript toward submission, reviewing current use cases, organizing reproducibility materials, and defining concrete action items and timelines.

### mzQC Manuscript Scope and Positioning Finalized

* The manuscript will serve as the primary community-facing introduction to mzQC.
* It is intended to:
  * Explain the motivation for standardized QC in biological mass spectrometry.
  * Demonstrate practical applications through multiple real-world use cases.
  * Encourage adoption across proteomics, metabolomics, lipidomics, and related domains.
* The manuscript will remain less technical than the mzQC specification and software library papers.
* Participants agreed to avoid framing mzQC as proteomics-specific.

### Current Use Cases Considered Strong Enough for Manuscript

The group agreed that the manuscript already contains a sufficiently broad and convincing set of use cases:

* Proteogenomics outlier detection using QuaMeter + PTXQC.
* Longitudinal system suitability monitoring using MSstatsQC.
* DIA reproducibility analysis using DIAuditor.
* Repository-scale metabolomics QC analysis (~800,000 runs).
* Lipidomics QC workflow and batch-effect analysis.

Potential additional sse case:

* GC-MS and LC-MS QC datasets from Steffen Neumann are being evaluated.
  * Interesting for demonstrating applicability outside LC-MS proteomics.
  * Current examples may be "too obvious" in terms of QC failure.
  * No commitment yet to include them in the manuscript.

### Emphasis on Reproducibility

A major decision was to fully consolidate all:

* Scripts
* Intermediate files
* mzQC exports
* Figures
* Workflow descriptions

into a shared repository structure.

Agreed workflow:

* Contributors will upload all materials into the shared Google Drive.
* Materials will later be cleaned and organized into GitHub repositories and reproducible workflows.
* The methods section will be written directly from the finalized workflows and scripts.

### Key Scientific Findings Highlighted

Proteogenomics Outlier Detection:

* QC metrics successfully detected known batch effects and electrospray instability.
* Identification-free QC metrics were shown to reveal issues masked by identification-based QC alone.

Repository-Scale Metabolomics:

* QC metrics from ~800,000 runs revealed clear clustering by acquisition strategy:
  * MS1-only
  * MS/MS
  * Positive/negative polarity modes
* Demonstrated utility for large-scale repository filtering and meta-analysis.

Lipidomics QC:

* Randomized block design and repeated QC samples provide a strong demonstration dataset.
* Focus will shift toward:
  * PCA-based batch-effect detection.
  * Coefficient of variation analysis.
  * Mapping existing metrics to mzQC CV terms.

## QC Metric Categorization and CV Updates

* The group revisited the proposed QC-metric categorization framework intended to make QC metrics more consistently and machine-readably annotated in the CV.
* Proposed dimensions included:
  * analytical dimension
  * workflow stage
  * information dependency
  * measurement scope
  * acquisition strategy
  * quality directionality
  * metric value type
* The biggest unresolved issue is redundancy between:
  * analytical dimension (e.g., chromatography, ionization, ion mobility);
  * workflow stage categories, which overlap heavily with the same concepts.
* The current CV hierarchy based on value type (single value, table, matrix) was considered semantically weak.
* The group agreed the system improves discoverability and machine reasoning but still needs refinement before broad adoption.

### Key Concerns Raised

* Many metrics naturally span multiple categories, making strict classification difficult.
* There is concern about:
  * combinatorial explosion of metrics.
  * ambiguity in assigning a "dominant" category.
  * ontology-browser limitations, since custom relationships are not well surfaced in standard CV viewers.
* Computational QC metrics (e.g., deconvolution effectiveness in top-down workflows) also need better representation in the model.
* Directionality ("higher is better") is not universally applicable and may depend on context or distributions.

### LLM-Assisted QC Metric Tool

* Fatemeh demonstrated a prototype web application that:
  * checks proposed QC metrics against existing CV terms;
  * flags duplicate/high-overlap metrics;
  * proposes classifications and relationships;
  * drafts concise CV-compatible names and definitions.
* The tool is intended as a human-assistance system, not automated CV generation.
* The group viewed this as a useful constrained application of LLMs.
* Faster/smaller models, embeddings, and knowledge-graph approaches may improve performance.

### CV Term Requests and Immediate Priority

* Dave emphasized that TDAuditor and DIAuditor manuscripts require stable CV accession numbers immediately.
* The group agreed:
  * accession numbers should be prioritized now;
  * full categorization relationships can be added later once the model stabilizes.
* Existing relationships drafted for new metrics should not be discarded but may remain provisional.

### Reuse of Existing CV Terms

* Discussion covered whether existing MS CV terms (e.g., mass resolving power) can be reused directly as QC metrics.
* Consensus leaned toward:
  * avoiding duplication of existing terms;
  * potentially adding QC-related relationships to existing MS terms instead.

### QC Tool Landscape Overview

* Fatemeh also presented an overview table of existing QC tools including:
  * acquisition modes;
  * data dependencies;
  * mzQC support;
  * platform and maintenance information.
* The table may later be mapped onto the proposed categorization framework to help users identify appropriate tools and metrics.

## mzQC Tools Ecosystem and Adoption

* Existing mzQC-supporting tools were reviewed:
  * MSQuality (Thomas Naake)
  * OpenMS
  * iDIA-QC
  * DDAuditor / DIAuditor
  * rmzQC, pymzQC, jmzQC
  * pmultiqc prototype support
* The group agreed that adoption is improving, but broader uptake requires:
  * Better developer documentation
  * Easier CV term submission workflows
  * Stable reference libraries
  * Outreach to external tool developers

### pymzQC Maintenance Concerns

* pymzQC currently provides the main semantic validation implementation for mzQC.
* Matthias, the primary maintainer, is no longer actively involved.
* Participants agreed the library is critical infrastructure and needs active maintenance support.
* Concerns were raised about maintainability and coding style, though the codebase is not considered too large to refactor if necessary.

### Repository-Scale QC and Visualization

* The group discussed how to apply mzQC at PRIDE/repository scale.
* Consensus emerged that:
  * Run-level metrics are more appropriate than dataset-level summaries.
  * ID-free metrics based directly on raw data are the most scalable approach.
  * Heterogeneity across PRIDE datasets remains a major challenge.
* Desired future capabilities include:
  * Longitudinal instrument monitoring
  * QC trend analysis over time
  * Linking QC metrics to maintenance/calibration events
  * Queryable QC metrics stored alongside repository data

### pmultiqc Integration Challenges

* The existing pmultiqc mzQC export pull request was reviewed conceptually.
* The implementation currently modifies ~1,400 lines across 21 files and was considered too tightly coupled and difficult to maintain.
* Participants questioned whether pmultiqc is the correct abstraction layer for generating mzQC.
* Alternative direction proposed:
  * Generate mzQC earlier in the workflow.
  * Use lightweight standalone visualization tools rather than deeply integrating into pmultiqc.

### ID-Free Metrics Work

* Timo Sachsenberg implemented ~100 ID-free metrics during the ELIXIR BioHackathon.
* Metrics include:
  * Spectral counts
  * Retention time ranges
  * Peak density summaries
  * TIC behavior
  * Charge distributions
  * Empty scans
  * Acquisition frequencies
* Current status:
  * Metrics still exist as a separate library.
  * Testing on larger PRIDE-scale datasets is pending.
  * Redundant metrics may later be removed.

### Visualization Strategy

* The group agreed that mzQC currently lacks mature "consumer" tools.
* A lightweight mzQC viewer/visualizer was discussed as an important next step.
* Potential features:
  * Longitudinal QC plots
  * Generic visualization based on metric categorization
  * Drag-and-drop mzQC visualization for non-programmers
* Participants noted that many useful visualizations can be generated directly from self-contained mzQC files without raw data access.

### Outreach and Community Engagement

* Personalized outreach to external tool developers was considered more effective than relying solely on publications.
* Commercial tools such as Spectronaut were discussed as valuable potential adopters.
* The group plans to contact actively maintained QC tool developers after the next preprint is released.

### Documentation and Training Materials

* Existing mzQC documentation was considered useful but fragmented.
* Main pain points:
  * CV term request process
  * Discoverability of developer resources
  * Lack of beginner-oriented workflows
* Nils shared CC BY-licensed educational slides from a metabolomics training day.
* The group agreed more training materials and example notebooks would help adoption.

## mzQC & mzTab-M Integration

### mzTab-M 2.1 Progress

* mzTab-M 2.1 is targeted for release later this year.
* Major schema redesign work is nearing completion, especially around:
  * study variable groups,
  * improved downstream processing support,
  * easier file generation.
* A new protocol section will be added to support mzTab-M ↔ ISA-Tab/MetaboLights interoperability.

### mzQC and mzTab-M Integration

* Consensus favored a **lightweight integration model**:
  * mzQC remains an auxiliary QC file linked from mzTab-M.
  * mzTab-M should remain self-contained as an archival/reporting format.
* Proposed integration mechanism:
  * use typed URI references in mzTab-M,
  * include CV terms and versioning for linked resources.
* Minimum interoperability requirement:
  * MS run identifiers must match between mzTab-M and mzQC.

### Study Variable Groups and QC Mapping

* Study variable groups in mzTab-M will provide grouping structure for assays/MS runs.
* mzQC set-level metrics can map onto these study variable groups.
* The relationship will not be enforced by schema validation but documented through guidance/examples.

**Outcome:** The group agreed to produce:
* a guidance document,
* example mzTab-M + mzQC files,
* a minimal illustrative dataset.

### Validation Architecture and Profiles

* Current validator limitations were discussed, especially around semantic validation.
* JSON Schema/OpenAPI alone are insufficient for expressing complex constraints.
* Open Policy Agent (OPA/Rego) was proposed for modular semantic validation.

### Version Migration and AI-Assisted Conversion

* The group discussed using AI-assisted tooling to:
  * migrate older mzTab-M versions,
  * generate compatibility transformations,
  * reduce maintenance burden for legacy readers.

### Lipidomics Data Standardization

* Significant discussion focused on poor lipidomics data deposition rates and incomplete metadata reporting.
* Many lipidomics datasets still exist only as supplementary tables without raw data deposition.
* Current lipidomics community efforts focus on:
  * harmonized metadata,
  * nomenclature normalization,
  * QC integration,
  * repository interoperability.
* A lipidomics aggregation platform is under development:
  * curated from published studies,
  * supports cross-study comparison,
  * integrates normalized units and lipid naming,
  * intended as a semantic analysis layer rather than a raw-data repository.
* mzTab-M is being used as the internal exchange/reporting format.
