## mzQC Specification 1.1

* The group discussed a limited mzQC 1.1 update, not a 2.0 release, while mzQC 1.0 manuscript work is still ongoing.
* Main candidate updates for mzQC 1.1 are:
  * Metric classification.
  * JSON Schema modernization.
  * Optional software settings in `analysisSoftware`.
  * CV/unit clarification.
  * Semantic validation and merging guidance.
  * Possible specification document format changes.

### JSON Schema

* mzQC currently uses JSON Schema draft-07.
* Updating to a newer stable schema version such as 2020-12 would not change mzQC file content, only validation/schema mechanics.
* The upgrade is non-trivial and should be reviewed carefully.
* **Action:** Revisit JSON Schema upgrade, potentially using an AI coding agent to assist.

### Software Settings

* Current mzQC provenance records which software was used but not the relevant settings.
* Missing settings can make metrics ambiguous, especially:
  * FDR thresholds.
  * Search parameters such as enzyme, missed cleavages, FASTA source, and filtering settings.
* The group favored adding optional `softwareSettings` under `analysisSoftware`.
* Preferred structure: simple JSON key–value object, not CV-backed parameters.
* Only relevant settings affecting metric interpretation should be recorded.
* **Decision:** Add optional lightweight `softwareSettings` to `analysisSoftware`.

### Matrix Metrics

* Matrix metrics currently lack row/column labels.
* The group struggled to identify a strong real-world matrix use case.
* Potential examples included DIA, ion mobility, LC–MS heat maps, and proteoform charge-state comparison matrices.
* Because the use case remains unclear, the group leaned toward leaving matrix unchanged for mzQC 1.1.
* A specification/CV inconsistency was found: table and matrix metric type accessions are shifted by one in the specification.
* **Action:** Fix table/matrix accession numbers in the specification.
* **Decision:** Do not extend matrix metrics for now unless a concrete use case emerges.

### Tables and Optional Columns

* Table metrics use CV accessions for columns.
* There may be a need for tables with dynamic or undefined additional columns, but the concrete use case was not available.
* **Action:** Revisit once the exporter use case is identified.

### Controlled Vocabulary and Units

* Some Unit Ontology terms are mirrored in PSI-MS CV.
* Loading both UO and PSI-MS caused duplicate-accession validation issues.
* **Action:** Clarify unit sourcing in the specification to avoid duplicate CV definitions.

### Semantic Validation

* The semantic validation list in the specification appears to match pymzQC.
* A GitHub "living document" could be useful but is not urgent.
* **Action:** Verify semantic validation checks remain synchronized with pymzQC.

### mzQC Merging

* Merging mzQC files can create conflicts in top-level metadata such as contact information and CV versions.
* pymzQC has an initial merge implementation, but its conflict-handling behavior needs checking.
* **Action:** Review Matthias's pymzQC merge implementation and document recommended merge behavior.

### Metric Classification

* Metric classification remains the main unresolved mzQC 1.1 issue.
* There is concern that classification may add implementation burden without clear perceived value for authors.
* **Action:** Resolve whether and how metric classification should be included before mzQC 1.1.

### Specification Format

* The group discussed moving the specification source from PDF/Google Docs toward GitHub-based Markdown or AsciiDoc.
* PDF could still be generated for formal tagged releases.
* **Likely direction:** Use GitHub for draft/review workflow and retain PDF as a release artifact.*

### Working Group Goals

Repository-Scale QC and pmultiqc:

* Major discussion focused on expanding QC from submission-time validation toward large-scale post-hoc QC of public repository data.
* pmultiqc was highlighted as a key initiative for:
  * Generating QC reports for PRIDE datasets.
  * Making dataset quality more transparent and reusable.
  * Enabling users to judge datasets before downloading/reanalyzing them.
* The group agreed repository-visible QC reports could encourage authors to discuss data quality more explicitly in publications.

ID-Free QC Infrastructure:

* Strong support for scalable ID-free QC workflows operating directly on raw files.
* Proposed workflow goals:
  * Compute standardized metrics across PRIDE datasets.
  * Export mzQC-compatible outputs.
  * Support DDA, DIA, TMT, and eventually top-down workflows.
* Consensus that the first implementation should focus on:
  * Simple, broadly applicable metrics.
  * DDA and DIA before more specialized workflows.
* Nextflow-based scalable infrastructure was discussed as the preferred implementation strategy.

Metric Standardization:

* The group emphasized the need for a minimal core metric set rather than overwhelming users with too many QC values.
* Suggested universally useful metrics included:
  * Number of MS1/MS2 scans.
  * Charge state distributions.
  * Retention time ranges.
  * Injection times.
* More specialized metrics can later be layered on top for advanced use cases.

Metadata and Dataset Classification:

* PRIDE is improving metadata annotation for:
  * TMT subtype detection.
  * DIA acquisition types.
  * Experimental modality classification.
* Participants discussed using QC-derived features to:
  * Automatically classify datasets.
  * Detect top-down experiments.
  * Identify calibration/blank runs.
  * Support searchable quality-aware repository interfaces.
